# AskProof｜AI验收官

🌐 语言：[English](README.md) | [中文](README.zh-CN.md)

> 别只让 AI 继续，先让它拿出证据。

AI 说“完成了”，不等于真的完成。

AskProof 帮非工程师多问一句：**凭什么说完成？**

AI 经常说：

- “已经完成。”
- “已经修复。”
- “现在应该可以了。”

但如果你不是工程师，你真正需要判断的是：

- 它到底改了什么？
- 有没有真的验证？
- 缺少哪些证据？
- 现在能不能继续？
- 下一句该怎么问 AI？

AskProof 是一个面向非工程师的 AI 验收与追问 Skill。

AI 会总结自己做了什么。

AskProof 帮你判断这件事能不能信。

它不替 AI 做事，也不替代测试工具。

它帮你看懂 AI 做了什么、判断它是否可信，并把“还是不行”“继续修”“你自己解决”这类低效反馈，
改写成能让 AI 拿出证据的高质量 Prompt。

**AI 说：**

> 已修复，现在应该可以了。

**普通用户可能会说：**

```text
请先不要继续扩展功能。请说明你改了哪些内容、用什么方式验证过、验证结果是什么、还有哪些内容没有验证。
```

## 你为什么需要 AskProof

AI 自带总结很有用，但总结不是验收证据。

AI 可能会说：

- “我已经修好了。”
- “我更新了实现。”
- “问题应该解决了。”
- “现在可以继续了。”

这些话可能只说明 AI 改了文件，不代表原问题真的消失，也不代表用户看到的页面、流程或结果已经可用。

AskProof 会帮你区分：

- AI 改了什么；
- AI 有没有证明它改对了；
- 哪些内容只是“已修改”，不是“已验证”；
- 当前能不能继续；
- 下一句该怎么追问 AI。

你不需要懂 Git、终端、日志、traceback、测试命令。

## AskProof 能帮你做什么

AskProof 帮助非工程师在 AI 说“完成了”之后，检查证据、发现未验证风险，并生成下一句该怎么问
AI 的高质量 Prompt。

它可以帮你：

- 判断“已完成 / 已修复 / 应该可以了”是否真的可信；
- 找出缺少哪些证据；
- 把“还是不行”“继续”“你自己解决”改写成可执行 Prompt；
- 在你不知道什么是日志时，引导你提供最低门槛证据；
- 用普通话解释 AI 的技术汇报；
- 在任务跑偏前提醒先收口、先验收；
- 在换对话、换模型、换平台前生成交接说明。

AskProof 不是又一个 AI coding 工具。它是给非工程师用的 AI 验收官。

## 适合谁使用

AskProof 适合正在用 AI 工具做事、但不一定懂工程细节的人：

- 非工程师
- 一人公司创业者
- 产品经理
- 运营
- 设计师
- 内容创作者
- 教育产品负责人
- 使用 Claude Code、Codex、Cursor、OpenClaw、国内 Agent 平台或其他 AI 工具的人

即使你不会看日志、不会跑测试、不会看 diff，也可以用 AskProof 让 AI 先把证据说清楚。

## V0.2 核心体验增强

AskProof V0.2 不新增前端、不做集成、不做自动测试执行器，而是强化一次完整 AI 验收体验。

- **Acceptance Brief｜验收简报**：把 AI 的“已完成”整理成当前问题、完成声明、已确认事实、已有证据、
  缺失证据、可信度、当前状态和下一步动作。
- **Evidence Type｜证据类型**：区分代码证据、日志证据、视觉证据、用户验收证据和可复现证据。
- **Same-Agent Verification Mode｜同一 Agent 自检模式**：当 Claude Code、Codex 或当前 Agent 能查看项目时，
  先检查已有项目证据，而不是只让用户提供日志。
- **自包含追问 Prompt**：下一句 Prompt 必须包含功能名、验收目标、需要的证据类型，以及“未验证前不要扩功能”。
- **Minimum Acceptance Path｜最小验收路径**：告诉用户最小应该打开哪里、看什么、什么算通过、失败后发什么给 AI。

## 不安装也能先试

你可以在任意 AI 对话里复制下面这段 Prompt，先体验 AskProof 的工作方式：

```text
请你扮演 AskProof，一个面向非工程师的 AI 验收与追问助手。

当我粘贴 AI Agent 的“已完成 / 已修复 / 现在应该可以了”回复时，不要默认它是真的完成。
请告诉我：
1. 当前属于什么场景；
2. 这句话是否可信；
3. 已确认的信息是什么；
4. 缺少哪些证据；
5. 现在继续的风险是什么；
6. 下一步最稳妥动作是什么；
7. 我应该复制哪一句话发回给 AI。

请用非工程师能听懂的语言。如果没有证据，必须标注“未验证”。

AI 说修好了，真的完成了吗？
```

如果 AskProof 生效，回答里应该出现：

- 完成可信度；
- 已确认信息；
- 缺少证据；
- 风险点；
- 下一句该怎么问 AI。

它不应该直接告诉你“是的，已经完成”。

安装后，你也可以直接用自然语言说：

- “我要新开对话，帮我生成交接说明”

## 核心功能

### 1. Done Check｜完成验收

当 AI 说“done / fixed / completed / should work now / 已完成 / 已修复 / 现在应该可以了”时，AskProof
会判断是否可信。

输出包括：

- 完成可信度：低 / 中 / 高
- 当前状态：已修改 / 已验证 / 未验证 / 有风险 / 可继续 / 不建议继续
- AI 提供了哪些证据
- 缺少哪些证据
- 用户应该如何验收
- 可直接复制给 AI 的追问 Prompt

### 2. Prompt Rescue｜低质量反馈救援

当你只会说：

- “还是不行”
- “继续”
- “你自己解决”
- “不要出现 bug”
- “重新改”
- “怎么又报错了”
- “still broken”
- “just fix it”

AskProof 会先解释为什么这句话低效，再把它改写成 AI 能执行的高质量 Prompt。

### 3. Evidence Ladder｜证据阶梯

AskProof 不会假设你懂日志。

当信息不足时，它会按最低门槛引导你提供证据：

- L0：只知道“不行”
- L1：能描述看到的现象
- L2：能发截图
- L3：能复制 AI 最后一段回复
- L4：能复制页面或终端里的红色错误文字
- L5：能提供终端日志 / Git diff / 测试结果

### 4. Explain｜解释 AI 做了什么

当你粘贴 AI 的技术汇报、开发汇报、执行结果或报错时，AskProof 会用普通话解释：

- AI 在说什么；
- 它实际做了什么；
- 这对项目有什么影响；
- 哪些内容已经确认；
- 哪些内容还没有验证；
- 下一步该怎么问 AI。

### 5. Drift Guard｜防跑偏

当你或 AI 不断加需求、优化、重构，但上一轮还没有验证时，AskProof 会提醒：

- 是否已经偏离原目标；
- 为什么现在不建议继续扩功能；
- 应该先验证什么；
- 是否需要生成阶段记录或 handoff；
- 是否应该先收口。

### 6. Handoff Memory｜交接记忆

当你要新开对话、换模型、换平台或交给下一个 AI 时，AskProof 会生成交接说明：

- 当前任务目标
- 已完成内容
- 已验证内容
- 未验证内容
- 当前风险
- 关键决策
- 下一轮可复制 Prompt
- 不要重复踩坑的事项

## 常见使用场景

AskProof 适合这些场景：

- Cursor 说修好了一个界面 bug，但没有截图或浏览器验证；
- Claude Code 说改了文件，但跳过了测试；
- Codex 运行了技术检查，但你不知道这能不能证明用户问题已修好；
- 国内 Agent 平台说“已修复并重新部署”，但没有说明验证结果；
- 产品经理只会反馈“还是不对”；
- 设计师觉得界面不对，但不知道怎么描述给工程型 AI；
- 运营要修自动化流程，但不想让 AI 一直盲改；
- 内容创作者或教育产品负责人要新开对话，需要保留上下文。

更多真实但虚构化场景见 [场景样例库](docs/example-gallery.md)。

## 示例 1：AI 说“修好了”

用户：

> AI 说：“已修复，现在应该可以了。”

AskProof：

```text
当前判断：Done Check｜完成验收
完成可信度：低
当前状态：AI 声称已修改，但未验证

已确认的信息：
- AI 声称问题已经修复。

未确认的信息：
- 改了哪些文件。
- 是否运行过项目。
- 原来的问题是否真的消失。
- 是否有截图、测试结果或人工验收记录。

风险点：
- 如果现在继续加功能，后面出问题时很难判断是旧问题还是新问题。

下一步最稳妥动作：
- 先让 AI 提供验证证据，再继续。

你现在最应该问 AI 的一句话：
请先不要继续扩展功能。请说明你改了哪些内容、用什么方式验证过、验证结果是什么、还有哪些内容没有验证。
```

## 示例 2：用户只会说“还是不行”

用户：

> 还是不行，你自己解决。

AskProof 改写：

```text
这句话低效的原因：
- AI 不知道你看到的失败现象是什么。
- AI 不知道上一次修改是否生效。
- AI 可能会继续盲目改代码，导致问题越来越难定位。

可直接复制给 AI：
请不要继续盲目改代码。请先按以下步骤排查：
1. 复述当前失败现象；
2. 说明你上一轮修改了哪些文件；
3. 判断问题可能发生在哪个环节；
4. 告诉我需要提供的最小证据；
5. 在没有新证据前，不要大范围重构；
6. 给出下一次修改后的验证方式。
```

## 示例 3：用户不知道怎么提供日志

AskProof 不会只说“请提供日志”。

它会说：

```text
你不用懂日志，先任选下面一种证据发回来：
1. 当前页面截图；
2. AI 最后一段“已完成/已修复”的回复；
3. 页面或终端里最后一行红色错误文字；
4. 你看到的异常现象，用普通话描述也可以。
```

## 示例 4：任务开始跑偏

用户：

> 登录修复还没验收，但顺便帮我加一个新看板，再重做设置页。

AskProof：

```text
当前判断：Drift Guard｜防跑偏
当前建议：现在不建议继续扩功能。

原因：
上一轮修改还没有验证证据。如果继续新增功能，后面很难判断问题来自哪里。

下一步最稳妥动作：
先让 AI 证明当前修改已经生效。

你现在最应该问 AI 的一句话：
请先暂停新增功能。请用清楚的证据验证登录修复已经生效，再总结哪些内容可以进入下一轮开发。
```

## 示例 5：新开对话前生成交接说明

```text
当前任务目标：
让虚构项目 TinyInvoice 支持导出 PDF 发票。

已完成内容：
- 添加了“导出 PDF”按钮。
- 接入了 PDF 生成函数。

已验证内容：
- 暂无验证证据。

未验证内容：
- PDF 是否真的下载成功。
- 金额和税费是否正确。

当前风险：
- AI 可能只完成了界面和函数连接，还没有验证完整流程。

下一轮可复制 Prompt：
请根据这份交接继续。不要先加新功能，请先验证“导出 PDF”按钮是否能下载 PDF，并确认发票金额正确。
```

## 和自动验证工具有什么区别

AskProof 不替代测试、CI、代码审查或自动验证工具。

它帮助非工程师在 AI 声称“完成”时，知道该如何追问、如何验收、如何避免被 AI 带着跑。

其他工具更偏向“让 Agent 提交验证证据”。

AskProof 更偏向“教用户向 Agent 要证据、理解证据、判断是否能信”。
