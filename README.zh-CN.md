# AskProof｜AI验收官

**别只让 AI 继续，先让它拿出证据。**

AI 经常说“已经完成”“已经修复”“现在应该可以了”。

但如果你不是工程师，你怎么判断它是不是真的完成？

AskProof 帮你看懂 AI 做了什么、缺少哪些证据、哪里还没验证，以及下一句该怎么问 AI。

AskProof 不替 AI 做事，而是帮你验收 AI、追问 AI、避免被 AI 带着跑。

## AskProof 是什么

AskProof 是一个面向非工程师的 AI 验收与追问 Skill。

它不做自动代码审查，也不替代测试框架、CI/CD、PR Review Bot 或 Agent 验证平台。

它的作用是：当 AI Agent 说“完成了 / 修好了 / 应该可以了”时，帮你判断这句话是否可信、证据是否足够、风险在哪里，以及下一句应该怎么问。

一句话：

AskProof 帮助非工程师在 AI 说“完成了”之后，检查证据、发现未验证风险，并生成下一句该怎么问 AI 的高质量 Prompt。

## 适合谁使用

AskProof 适合正在用 AI 工具做事、但不一定懂工程细节的人：

- 非工程师
- 一人公司创业者
- 产品经理
- 运营
- 设计师
- 内容创作者
- 教育产品负责人
- 使用 Claude Code、Codex、Cursor、OpenClaw、国内 Agent 平台或其他 AI 工具的人

你不需要懂 Git、终端、日志、traceback、测试命令。

## 为什么 AI 自带总结还不够

AI 很容易给出听起来很确定的总结：

- “已修复。”
- “现在应该可以了。”
- “问题已经解决。”
- “我已经更新代码，可以继续了。”

这些话只能说明 AI 声称自己做了事，不等于结果已经验证。

AskProof 会帮你区分：

- AI 改了什么；
- AI 有没有证明它改对了；
- 哪些内容只是“已修改”，不是“已验证”；
- 当前能不能继续；
- 下一句该怎么追问 AI。

## AskProof 和自动验证工具有什么区别

AskProof 不替代测试、CI、代码审查或自动验证工具。

它帮助非工程师在 AI 声称“完成”时，知道该如何追问、如何验收、如何避免被 AI 带着跑。

其他工具更偏向“让 Agent 提交验证证据”。

AskProof 更偏向“教用户向 Agent 要证据、理解证据、判断是否能信”。

## 核心功能

### 1. Done Check｜完成验收

当 AI 说“done / fixed / completed / should work now / 已完成 / 已修复 / 现在应该可以了”时，AskProof 会判断是否可信。

输出包括：

- 完成可信度：低 / 中 / 高
- 当前状态：已修改 / 已验证 / 未验证 / 有风险 / 可继续 / 不建议继续
- AI 提供了哪些证据
- 缺少哪些证据
- 用户应该如何验收
- 可直接复制给 AI 的追问 Prompt

### 2. Prompt Rescue｜低质量反馈救援

当你只会说：

- “还是不行”
- “继续”
- “你自己解决”
- “不要出现 bug”
- “重新改”
- “怎么又报错了”
- “still broken”
- “just fix it”

AskProof 会先解释为什么这句话低效，再把它改写成 AI 能执行的高质量 Prompt。

### 3. Evidence Ladder｜证据阶梯

AskProof 不会假设你懂日志。

当信息不足时，它会按最低门槛引导你提供证据：

- L0：只知道“不行”
- L1：能描述看到的现象
- L2：能发截图
- L3：能复制 AI 最后一段回复
- L4：能复制页面或终端里的红色错误文字
- L5：能提供终端日志 / Git diff / 测试结果

### 4. Explain｜解释 AI 做了什么

当你粘贴 AI 的技术汇报、开发汇报、执行结果或报错时，AskProof 会用普通话解释：

- AI 在说什么；
- 它实际做了什么；
- 这对项目有什么影响；
- 哪些内容已经确认；
- 哪些内容还没有验证；
- 下一步该怎么问 AI。

### 5. Drift Guard｜防跑偏

当你或 AI 不断加需求、优化、重构，但上一轮还没有验证时，AskProof 会提醒：

- 是否已经偏离原目标；
- 为什么现在不建议继续扩功能；
- 应该先验证什么；
- 是否需要生成阶段记录或 handoff；
- 是否应该先收口。

### 6. Handoff Memory｜交接记忆

当你要新开对话、换模型、换平台或交给下一个 AI 时，AskProof 会生成交接说明：

- 当前任务目标
- 已完成内容
- 已验证内容
- 未验证内容
- 当前风险
- 关键决策
- 下一轮可复制 Prompt
- 不要重复踩坑的事项

## 不用记命令的快速开始

你可以直接说自然语言：

- “AI 说修好了，真的完成了吗？”
- “还是不行”
- “我看不懂这个报错”
- “帮我把这句话改成更有效的 Prompt”
- “我要新开对话，帮我生成交接说明”
- “这轮到底做了什么？”
- “现在我该让 AI 做什么？”

高级用户也可以使用：

- `/askproof done-check`
- `/askproof prompt-rescue`
- `/askproof evidence`
- `/askproof explain`
- `/askproof drift-guard`
- `/askproof handoff`

## 示例：AI 说“修好了”

用户：

> AI 说：“已修复，现在应该可以了。”

AskProof：

```text
当前判断：Done Check｜完成验收
完成可信度：低
当前状态：AI 声称已修改，但未验证

已确认的信息：
- AI 声称问题已经修复。

未确认的信息：
- 改了哪些文件。
- 是否运行过项目。
- 原来的问题是否真的消失。
- 是否有截图、测试结果或人工验收记录。

风险点：
- 如果现在继续加功能，后面出问题时很难判断是旧问题还是新问题。

下一步最稳妥动作：
- 先让 AI 提供验证证据，再继续。

你现在最应该问 AI 的一句话：
请先不要继续扩展功能。请说明你改了哪些内容、用什么方式验证过、验证结果是什么、还有哪些内容没有验证。
```

## 示例：用户只会说“还是不行”

用户：

> 还是不行，你自己解决。

AskProof 改写：

```text
这句话低效的原因：
- AI 不知道你看到的失败现象是什么。
- AI 不知道上一次修改是否生效。
- AI 可能会继续盲目改代码，导致问题越来越难定位。

可直接复制给 AI：
请不要继续盲目改代码。请先按以下步骤排查：
1. 复述当前失败现象；
2. 说明你上一轮修改了哪些文件；
3. 判断问题可能发生在哪个环节；
4. 告诉我需要提供的最小证据；
5. 在没有新证据前，不要大范围重构；
6. 给出下一次修改后的验证方式。
```

## 示例：用户不知道怎么提供日志

AskProof 不会只说“请提供日志”。

它会说：

```text
你不用懂日志，先任选下面一种证据发回来：
1. 当前页面截图；
2. AI 最后一段“已完成/已修复”的回复；
3. 页面或终端里最后一行红色错误文字；
4. 你看到的异常现象，用普通话描述也可以。
```

## 示例：新开对话前生成交接说明

```text
当前任务目标：
让虚构项目 TinyInvoice 支持导出 PDF 发票。

已完成内容：
- 添加了“导出 PDF”按钮。
- 接入了 PDF 生成函数。

已验证内容：
- 暂无验证证据。

未验证内容：
- PDF 是否真的下载成功。
- 金额和税费是否正确。

当前风险：
- AI 可能只完成了界面和函数连接，还没有验证完整流程。

下一轮可复制 Prompt：
请根据这份交接继续。不要先加新功能，请先验证“导出 PDF”按钮是否能下载 PDF，并确认发票金额正确。
```
